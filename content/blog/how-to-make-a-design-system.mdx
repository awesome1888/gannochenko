## Starter points

20+ projects, lots of people...

### Taking to designers and other devs

Speak same language. Establish common terminology.

### Tech aspects

Start with 0.0.0
Don't use ^ in semver
Describe 3-level token system, draw a picture.

Token system is essentially a contract between designers and devs. Initially, it is a subject to negotiations, done in several iterations. Dark theme should also follow the contract.

Describe the minimum set of components. Describe color and typography naming.

### Testing

Loki может тестить только статичные состояния - его используем как альтернативу react testing library snapshots
Jest более ориентирован на тест логики - тестим API компонентов, хелперные функции, темы, комбинации параметров компонента, вызовы callbacks и тд
E2E тестирование с TestCafe или Cypress - натравливаем на storybook и тестируем сценарии, комплексные комбинации компонентов, как они взаимодействуют и т.д. Кроме того, можно тестить mouse events, чего jest не позволяет.

### Migration

No big bang please! (Too long, will stale all other work).
Rolling per-component update instead. Small PRs, high involvement.
Maybe need to apply a theme to adjust new components to the old ones. Then, theming gets removed and we have a fully migrated application.

Build basic components and only then offer collaboration, otherwise they will struggling with having more complex things when there is still no basic stuff done.
Постоянно информируй о прогрессе и о планах на грядущие компоненты.

Об именовании токенов первого уровня: мы пытались придать именам семантику, называть например spacing как spacingSingle, spacingDouble или spacingMD, spacingXL, ... но это плохо работало, поскольку если нам нужно больше градаций или значения между, то у нас будет проблема. Поэтому решили остановиться на числах с шагом.

### Architecture

react hook based library is a fresh way of looking at things, but we decided to go more classic way, because

1. our design is not separable from the logic we provide, that is why shipping hooks and elements separately into an application space is not an option
2. we wanted to make the components as simple for an end user as possible
3. high visual customisation is not needed, and in fact it is even discouraged (as we want the same look and feel across all the apps)
